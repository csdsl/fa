# 面试模拟总结 - 第一天

## Q1: 为什么用AI Agent做结账，传统工作流引擎不行吗？

**参考答案：**

金蝶云星空的期末结账是多模块、多步骤、有依赖关系的复杂流程（凭证检查→调汇→计提折旧→费用摊销→损益结转→对账→正式结账）。

传统工作流引擎的痛点：
- 不同企业结账流程有差异，每种都要画流程图维护
- 异常处理只能走预设分支，不够灵活
- 用户想跳过某步或重试某步，改起来麻烦

Agent的优势：
- 通过Prompt定义规则，不同客户只需调整Prompt而非重画流程
- Agent能根据当前状态动态决策，遇到异常自主判断是跳过、重试还是告警
- 支持自然语言交互，用户可以说"跳过调汇直接计提"

本质区别：传统工作流适合确定性流程，Agent适合有判断、有变化的半结构化流程。

## Q2: KWork的整体技术架构是怎样的？

**参考答案：**

四层架构：

1. 交互层：结账面板 + 对话框，支持一键自动结账和对话式交互
2. Agent编排层：基于Spring AI的ChatClient，采用ReAct模式（推理→调用工具→观察结果→决定下一步）
3. 工具层：通过MCP协议注册各结账工具（check_voucher_tool、depreciation_tool、profit_loss_carry_tool、period_close_tool等）
4. 业务服务层：Tool调用已有的ERP业务服务API，保持事务性和权限控制不变

调用链路：用户触发 → Controller → ChatClient发起对话 → LLM推理决定调用哪个Tool → 通过MCP调用结账服务 → 返回结果 → LLM判断下一步 → 循环直到完成

## Q3: MCP解决什么问题？和Function Calling的区别？

**参考答案：**

MCP在我们场景里解决的问题：
- 金蝶云星空是多模块ERP，后续要扩展到对账、报表、资产盘点等场景
- 用MCP后，每个业务模块作为独立MCP Server暴露Tool，Agent作为Client动态发现和调用
- 新增场景只需部署新的MCP Server，Agent不用改代码

和Function Calling的区别：
- Function Calling：Tool和Agent在同一进程内，编译时绑定，简单直接，适合工具少且固定的场景
- MCP：Tool和Agent可跨进程/跨服务，标准协议通信，支持动态发现、版本管理、权限隔离，适合平台化多团队协作

一句话：Function Calling是"内聚"，MCP是"解耦"。选MCP因为金蝶是平台型产品，需要各模块独立演进。

## Q4: 线程池参数怎么设定的？

**参考答案：**

1. 分析任务类型：资产折旧计算是CPU密集+少量IO（读卡片数据、写结果），偏计算型

2. 初始参数（理论依据）：
   - corePoolSize：CPU核数+1 = 9（8核服务器）
   - maximumPoolSize：核数×2 = 16
   - 队列：LinkedBlockingQueue，容量500（50万/每批1000）
   - 拒绝策略：CallerRunsPolicy

3. 压测调优：
   - 发现瓶颈在数据库IO而非纯CPU
   - corePoolSize调大到12-15，让更多线程并行等待IO
   - 批次大小从1000调到2000，减少数据库交互次数

4. 最终参数：核心12，最大20，队列256
5. 为什么不用ForkJoinPool：我们是批量独立任务，没有父子关系，ThreadPoolExecutor更直观可控

## Q5: RabbitMQ怎么保证消息可靠性？

**参考答案：**

三个环节分别保障：

1. 生产者丢失（发送端→MQ）：
   - 开启Publisher Confirm，消息到达Exchange后回调确认
   - confirm失败记录到本地消息表，定时重试
   - 配Return回调感知路由失败

2. MQ自身丢失（宕机）：
   - Exchange和Queue设置durable持久化
   - 消息deliveryMode=2持久化
   - 镜像队列集群模式

3. 消费端丢失（MQ→消费者）：
   - 关闭自动ACK，手动ACK
   - 处理失败消息重新入队
   - 消费端幂等处理（消息ID + Redis SETNX）

4. 兜底：本地消息表 + 每日对账

## Q6: 短信重复消费怎么防？

**参考答案：**

三层防护：

1. Redis快速去重：消费者拿到消息后用Message ID做SETNX（过期24h），SET失败直接ACK丢弃
2. 数据库状态表兜底：短信发送记录表加唯一索引(用户ID+业务类型+业务流水号)，插入成功才调短信网关
3. 短信网关层面：对接的网关支持相同手机号+相同内容N分钟内去重

为什么不只用数据库：Redis判重O(1)毫秒级返回，大部分重复在Redis层拦掉，不打数据库。

## Q7: 分布式多维度限流方案设计

**参考答案：**

整体方案：网关层统一拦截 + Redis集中式计数 + 多维度Key设计

三层架构：
1. Nginx层（IP级粗粒度）：limit_req_zone按IP限流，防攻击和爬虫
2. API网关层（接口级）：Redis + Lua脚本实现滑动窗口，Key=`rate_limit:{接口路径}:{时间窗口}`
3. 业务层（用户级）：Key=`rate_limit:{userId}:{接口路径}:{时间窗口}`

核心实现：Redis Sorted Set + Lua脚本（滑动窗口算法），保证原子性

限流规则配置化（Nacos），支持动态调整。被限流返回429 + Retry-After。

高可用：Redis集群，Redis不可用时降级为本地Guava RateLimiter。

算法选择：银行/ERP用滑动窗口（严格控速），秒杀用令牌桶（允许突发）。

## Q8: ELK日志架构，ES查询优化和冷热分离

**参考答案：**

数据量：日均2-3亿条，每天200-300GB，保留热7天/温30天/冷90天。

冷热分离落地：
- 索引按天创建：`log-service-2023.05.13`
- Hot节点（SSD）：最近7天，承担写入和高频查询
- Warm节点（HDD）：7-30天，只读
- Cold节点（大容量HDD）：30-90天，极少查询
- ILM策略自动迁移：7天rollover到Warm + force merge，30天到Cold，90天删除

其他优化：
- 日志分级采样（ERROR全量，INFO 10%采样）
- 异步写入（Log4j2 AsyncAppender）
- Filebeat → Kafka → Logstash → ES（Kafka做缓冲削峰）

效果：热数据查询从5-8秒降到1秒内，存储成本降40%。

## Q9: 资产管理模块分表方案

**参考答案：**

方案：按租户ID分库，库内按资产类别分表。

为什么按组织ID分库：
- ERP查询100%带组织ID条件
- 折旧计算按组织批量处理，分库后是单库操作，事务简单

库内分表：
- 资产卡片表按资产类别水平分表（固定资产、无形资产、长期待摊）
- 折旧明细表按年月分区（增长最快，查询都带期间条件）

中间件：ShardingSphere-JDBC

注意事项：
- 跨组织报表走异步汇总
- 全局唯一ID用雪花算法
- DDL变更需批量执行

核心原则：分片Key跟着业务最高频查询条件走。

## Q10: 银行还款场景的分布式事务方案

**参考答案：**

推荐方案：本地消息表 + 最终一致性（而非TCC）

原因：还款是单向推进流程（扣款成功→更新账单→发通知），不需要全局回滚。TCC开发成本高且外部渠道不一定支持Cancel。

流转：
1. 账户服务开启本地事务：扣余额 + 写本地消息表（同一事务）
2. 定时任务/投递器扫描消息表，发到RabbitMQ
3. 信用卡服务消费消息，减欠款更新账单
4. 通知服务消费消息，发短信

TCC适用场景：跨行实时转账（A扣B加必须原子性）
最终一致性适用场景：还款（扣款成功后续步骤可异步）

## Q11: 多线程结果收集 + 异常处理

**参考答案：**

用CompletableFuture提交任务和收集结果：
1. 数据分片（每批2000条）
2. CompletableFuture.supplyAsync提交到线程池
3. CompletableFuture.allOf等待全部完成
4. 逐个get()区分成功和失败
5. 失败批次记录后重试（最多3次）

异常处理策略：部分失败不影响整体。资产折旧按卡片独立计算，A失败不影响B。成功的正常入库，失败的标记"计算异常"人工处理。

为什么用CompletableFuture而非Future：支持allOf组合、exceptionally异常处理、thenApply链式处理，且可指定自定义线程池。

## Q12: 5000租户批量生成报表的架构设计

**参考答案：**

先算账：5000×30秒=41.7小时串行，6小时完成需至少8并发，设计16并发留余量。

架构：
1. 调度层：XXL-Job每月1号触发，主任务拆分5000个子任务写入任务表
2. 分发层：数据库任务表（status/worker_id/retry_count），多Worker抢占式消费（SELECT FOR UPDATE或Redis队列）
3. 执行层：4台机器×4线程=16并发，每个Worker取任务→计算→生成文件→更新状态
4. 失败重试：FAILED且retry<3自动重置为PENDING；Worker心跳超时5分钟自动恢复
5. 进度监控：按status统计，接入Grafana看板，失败率超阈值告警
6. 资源保护：数据库连接池限制、大租户优先、限流保护在线业务

核心思维：架构设计题先量化分析（算账），再给方案。
