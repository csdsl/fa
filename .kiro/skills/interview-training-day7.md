## 题目1：Redis缓存架构设计与一致性

### 面试官问题
你们的Redis缓存架构是怎么设计的？缓存了什么数据？遇到过缓存和数据库不一致的情况吗？怎么解决的？

### 标准答案（口述版）

Redis Cluster模式，6节点（3主3从），每个主节点分配固定slot范围，单节点32GB。选Cluster而不是哨兵，因为银行App百万级用户，单节点内存和QPS有瓶颈。

缓存了四类数据：
1. **验证码**：Key=`sms:{phone}:{bizType}`，TTL 5分钟，String类型
2. **用户Session/Token**：Key=`session:{userId}`，TTL 30分钟，Hash存基本信息
3. **热点账户信息**：Key=`card:{cardNo}:info`，TTL 10分钟
4. **账单列表**：Key=`bill:{userId}:{month}`，TTL 1小时

一致性保障——Cache-Aside策略：
- 读：先查缓存→miss查DB→回填缓存
- 写：**先更新DB，再删缓存**（不是更新缓存）

为什么这个顺序：先删缓存再更新DB有并发问题——线程A删缓存→线程B读miss回填旧值→线程A更新DB。结果缓存是旧值。

兜底：Canal监听MySQL Binlog异步删缓存。应用层直接删为主路径 + Canal异步兜底 + 积压超1000条告警。

### 回答要点
1. 集群具体方案（几个节点、什么模式）
2. 缓存的具体业务数据（Key设计+TTL）
3. Cache-Aside的读写顺序
4. 为什么"先更新DB再删缓存"（并发问题分析）
5. Canal兜底 + 踩坑经验

## 题目2：缓存雪崩/击穿防护

### 面试官问题
月初大量用户查账单，缓存Key同时过期了怎么办？

### 标准答案（口述版）

四层防护：

**1. 预防：预热+随机过期**
- 月初凌晨定时任务批量预热活跃用户Top 10万的账单缓存
- 所有Key的TTL = 基础1小时 + 随机0-10分钟

**2. 击穿防护：Redisson锁+逻辑过期**
- Value里存`{"data":..., "expireAt": timestamp}`，物理Key不设TTL
- 逻辑过期后tryLock，抢到的异步刷新，抢不到的返回旧数据
- 用户永远能拿到数据，不会穿透DB

**3. 多级缓存：Caffeine L1 + Redis L2**
- Caffeine本地缓存TTL 10秒，10秒内重复请求直连内存

**4. 兜底：限流降级**
- Sentinel保护DB，降级返回上一期快照+"数据加载中"

### 回答要点
1. 区分雪崩（大批同时过期）vs 击穿（单热点Key过期）
2. 预热是最有效手段（可预测热点）
3. 逻辑过期的具体实现
4. Caffeine的TTL要短（10秒）否则一致性更差

## 题目3：逻辑过期的并发细节

### 面试官问题
1000个请求同时发现缓存逻辑过期了，999个线程都会抢锁。是排队等还是返回旧数据？CFO看到昨天的账单怎么解释？

### 标准答案（口述版）

**选择：直接返回旧数据，不排队等。**

"旧数据"不是昨天的——而是过期了不到100ms的数据：
```
T+0ms:     第1个请求tryLock成功，异步刷新
T+0ms:     第2-1000个请求tryLock失败，返回旧数据
T+80ms:    刷新完成，缓存更新为最新
T+80ms后:  所有新请求拿到最新数据
```

防护：逻辑过期1小时 + 物理兜底TTL 2小时。如果刷新一直失败，最多返回2小时内旧数据，之后物理过期走正常miss路径。

核心取舍：**宁可返回99.9%正确的数据让用户立即看到，也不让999人转圈等100ms。**

注意：余额不能用这个方案（必须秒级一致），用短TTL 10秒 + 主动失效（扣款后del key）。

### 回答要点
1. 选返回旧数据，不等待
2. "旧"只旧了不到100ms，不是一天
3. 物理TTL做硬兜底
4. 区分什么数据能用逻辑过期（账单可以，余额不行）

## 题目4：KWork反结账处理

### 面试官问题
用户说"帮我把上个月的结账撤销了重新来"，Agent能做吗？怎么处理？

### 标准答案（口述版）

Agent能做，但走"确认+反结账"流程。

**第一步：意图澄清**
Agent确认哪个组织、哪个期间、整体反结账还是部分撤销。

**第二步：前置条件校验**
调`check_reverse_close_tool`：
- 下一期是否已有凭证？有则阻断（跨期数据混乱）
- 用户是否有反结账权限？（通常财务主管以上）
- 合并报表/税务申报是否已取数？有依赖则拒绝

**第三步：二次确认后逆序执行**
输出执行计划要求确认后按逆序执行：
- reverse_period_close_tool：打开期间
- reverse_profit_carry_tool：冲销损益结转（生成红字凭证）
- 折旧状态重置为"可重算"

**关键设计原则**：
- 高风险操作必须人工确认
- 不"删除"数据，而是生成反向凭证保持审计轨迹
- Agent有权拒绝用户指令（权限不足/前置条件不满足）

### 回答要点
1. Agent能做但要确认+校验
2. 前置条件校验（3个维度）
3. 逆序执行
4. 不删除，用反向凭证
5. Agent能拒绝——不是无脑执行器

## 题目5：多租户隔离防突破

### 面试官问题
如果开发人员写原生SQL绕过了MyBatis拦截器查到其他租户数据，你怎么防？

### 标准答案（口述版）

四层纵深防御：

**1. MyBatis拦截器自动注入（覆盖95%）**
StatementHandler的prepare阶段改写SQL，自动追加WHERE tenant_id条件。

**2. 数据库视图/session变量兜底（防绕过）**
```java
conn.createStatement().execute("SET @current_tenant = '" + TenantContext.get() + "'");
```
核心表创建视图`WHERE tenant_id = @current_tenant`，开发人员操作视图不操作原表。

**3. 静态扫描+CI卡点（编译时拦截）**
- SonarQube自定义规则：核心表SQL不带tenant_id标Critical
- JdbcTemplate/原生SQL必须架构组审批
- CI流水线卡住不让合并

**4. 运行时审计+告警（事后发现）**
- 定期扫描SQL日志检测不带tenant_id的查询
- API返回结果抽样校验tenant_id一致性
- 发现异常立即告警+熔断

踩坑经验：新同事用JdbcTemplate忘加tenant_id，SonarQube拦住了。之后加了更严卡点：应用账号只能访问视图，原表只有DBA能直接访问。

核心原则：**不相信人，相信机制。**

### 回答要点
1. 四层：拦截器→视图→静态扫描→运行时审计
2. 真实踩坑案例
3. 金句："不相信人，相信机制"

## 题目6：LLM调用成本控制

### 面试官问题
大模型按token计费，怎么控制成本？租户疯狂调用怎么限制？有没有监控和预算管理？

### 标准答案（口述版）

四个层面：

**1. 模型路由（省钱核心）**
按任务复杂度分级：
- 结账编排（多步推理+工具调用）：qwen-plus
- 简单查询（"到哪一步了"）：qwen-turbo（便宜4倍）
- 意图分类：qwen-lite（最便宜，做预判）

先用lite做意图分类（1%成本），再路由到对应模型。

**2. 限流配额**
三维度：
- 全局QPS上限200
- 单用户每分钟最多10次
- 租户月度Token配额100万token（约5000次对话），超额降级或提示

配额存Redis：`token_usage:{tenantId}:{month}`，INCRBY原子累加。

**3. 缓存复用**
- 语义缓存：向量相似度>0.95直接返回历史答案（命中率约30%）
- Prompt精简：System Prompt从2000 token压缩到800 token

**4. 可观测性**
每次调用记录模型/input_tokens/output_tokens/耗时/是否命中缓存。Grafana看板+单租户日消耗超阈值告警。

效果：平均调用成本降低60%，月均从2万降到8千。

### 回答要点
1. 模型路由是最大省钱点
2. 限流三维度（全局/用户/租户月配额）
3. 语义缓存减少调用次数
4. 量化效果：降60%

## 题目7：Agent上下文窗口膨胀控制

### 面试官问题
对话越来越长token越来越大，怎么控制上下文窗口膨胀？

### 标准答案（口述版）

三层策略：

**1. 结构化状态替代对话记忆（核心策略）**
结账进度不靠对话历史——Agent每次推理前调`status_query_tool`获取真实状态。即使对话截断也不会"忘记"之前做了什么。
关键决策：**业务状态外置到数据库，不靠LLM上下文窗口记忆。**

**2. 滑动窗口+摘要压缩**
- 保留：System Prompt(800token) + 最近3轮 + 当前Tool结果
- 压缩：超过3轮的历史→用LLM生成摘要(~200token)注入上下文
- 丢弃：Tool中间过程日志（只保留最终结果）

**3. 按角色裁剪**
- System Prompt：永远保留
- Tool返回：只留最近一次完整，之前只留"步骤X：成功"
- 用户消息：保留最近2条
- Agent的Thought：完全丢弃

效果：7步结账约10轮对话，上下文始终控制在3000-4000 token。不控制会膨胀到8000-10000。

### 回答要点
1. 核心策略：状态外置到DB，不靠对话记忆
2. 滑动窗口+摘要压缩
3. 按消息角色区分重要性裁剪
4. 给出具体数字（3000-4000 token）

## 题目8：HashMap多线程问题与ConcurrentHashMap

### 面试官问题
HashMap多线程下什么问题？ConcurrentHashMap怎么解决？JDK7 vs JDK8区别？

### 标准答案（口述版）

**HashMap多线程问题（3个）：**
1. **数据丢失**：两线程put同一桶位，后者覆盖前者
2. **JDK7扩容死循环**：头插法transfer形成链表环，get时CPU 100%。JDK8改尾插法解决了死循环（但仍有数据不一致）
3. **size不准确**：没有volatile/原子操作

**ConcurrentHashMap JDK7 vs JDK8：**

| 维度 | JDK7 | JDK8 |
|------|------|------|
| 数据结构 | Segment[] + HashEntry[] + 链表 | Node[] + 链表/红黑树 |
| 锁粒度 | Segment级（默认16段，最多16并发） | Node级（锁单个桶头节点） |
| 锁实现 | ReentrantLock | CAS + synchronized |
| 读操作 | 不加锁（volatile） | 不加锁（volatile+Unsafe） |
| 扩容 | 段内扩容 | 多线程协同扩容（transferIndex分段领取） |

**JDK8核心改进：**
1. 锁粒度：段级→桶级，并发度从固定16提升到数组大小
2. CAS+synchronized配合：桶空时CAS无锁写入，非空时synchronized锁头节点
3. 多线程协同扩容：其他线程put时发现扩容中会帮忙迁移
4. 红黑树：链表>8转红黑树，O(n)→O(logn)

一句话：JDK7是"粗粒度分段锁"，JDK8是"细粒度CAS+synchronized+协同扩容"。

### 回答要点
1. HashMap问题：数据丢失+JDK7死循环+size不准
2. JDK7 vs JDK8用对比表
3. 核心差异：锁粒度从段级→桶级
4. CAS+synchronized的分工（空桶CAS，非空synchronized）
5. 协同扩容是加分项
