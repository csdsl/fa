# KWork智能结账智能体 - 面试问答手册

## Q1: 结账流程具体包含哪些步骤？为什么需要用智能体来做，传统工作流引擎不行吗？

**答：**

金蝶云星空的期末结账是一个多模块、多步骤、有依赖关系的复杂流程。典型的结账流程包括：

1. 凭证检查（是否有未过账凭证）
2. 期末调汇（外币账户汇率调整）
3. 计提折旧（固定资产模块）
4. 费用摊销（待摊费用分摊）
5. 损益结转（收入费用结转到本年利润）
6. 期末对账（各模块数据核对）
7. 正式结账（关闭当期，开启下期）

传统工作流引擎（比如Activiti/Flowable）当然能做，但有几个痛点：

- 结账步骤的顺序不是完全固定的，不同企业、不同行业的结账流程有差异，传统工作流需要为每种情况画流程图，维护成本高
- 结账过程中经常出现异常（比如某张凭证有问题），传统工作流只能走预设的异常分支，不够灵活
- 用户经常需要"跳过某步"或"重试某步"，传统工作流改起来很麻烦

用Agent的优势在于：
- 通过Prompt定义结账规则和约束，不同客户只需要调整Prompt而不是重画流程
- Agent能根据当前状态动态决策下一步做什么，遇到异常可以自主判断是跳过、重试还是告警
- 用户可以用自然语言交互，比如"帮我跳过调汇直接计提折旧"，Agent能理解并执行

本质上，传统工作流适合确定性流程，Agent适合有判断、有变化的半结构化流程。结账恰好是后者。

## Q2: KWork的整体架构是怎样的？

**答：**

KWork的架构分为四层：

**交互层**：前端提供结账面板，用户可以一键触发自动结账，也可以通过对话框与Agent交互（比如询问当前进度、手动干预某步骤）

**Agent编排层**：这是核心，采用ReAct模式（Reasoning + Acting）。Agent接收到结账指令后，会：
- 先推理当前应该执行哪一步（Thought）
- 调用对应的工具执行（Action）
- 观察执行结果（Observation）
- 决定下一步动作

**工具层（Tools）**：每个结账步骤封装为一个Tool，包括：
- `check_voucher_tool` — 检查未过账凭证
- `depreciation_tool` — 计提折旧
- `exchange_adjustment_tool` — 期末调汇
- `profit_loss_carry_tool` — 损益结转
- `period_close_tool` — 正式结账
- `status_query_tool` — 查询当前结账状态

**业务服务层**：Tool不直接操作数据库，而是调用已有的ERP业务服务API。这样Agent层和业务逻辑解耦，业务服务的事务性、权限控制等都保持不变。

LLM选型上，我们用的是金蝶内部接入的大模型服务（对接了多个模型，支持切换），通过统一的API网关调用。

## Q3: Agent的工具（Tool）粒度怎么定的？

**答：**

工具粒度的设计原则是"一个Tool对应一个业务动作的最小完整单元"。

具体来说：
- 太细不行：比如把"查询凭证列表"和"判断是否有未过账"拆成两个Tool，Agent需要多次调用才能完成一个判断，增加了LLM推理次数和出错概率
- 太粗不行：比如把"检查+计提+结转"合成一个Tool，那Agent就失去了灵活编排的能力

我们的原则是：每个Tool执行完后，返回一个明确的状态（成功/失败/需要人工介入），Agent基于这个状态决策下一步。

另外，每个Tool都有清晰的入参和出参描述（JSON Schema），这样LLM能准确理解什么时候该调用哪个Tool、传什么参数。

## Q4: 多步骤结账任务的编排逻辑是怎么实现的？ReAct还是DAG？

**答：**

我们用的是ReAct模式，但加了约束。

纯ReAct的问题是Agent可能"乱跳"，比如还没检查凭证就直接结账。所以我们在Prompt中定义了结账步骤的依赖关系和前置条件：

```
结账规则：
1. 正式结账前必须完成损益结转
2. 损益结转前必须完成计提折旧和费用摊销
3. 计提折旧前必须完成凭证检查
4. 如果凭证检查发现未过账凭证，必须先处理完才能继续
```

Agent每次决策时，会参考这些规则。同时每个Tool返回结果中会带上"当前已完成步骤"的状态信息，Agent据此判断下一步。

为什么不用纯DAG？因为实际结账过程中有很多"意外"：
- 某步失败了，用户说"跳过这步继续"
- 用户中途说"先帮我看看应收应付的数据对不对"
- 某些步骤在特定条件下可以并行

这些动态决策用DAG很难表达，ReAct + 规则约束是更好的平衡。

## Q5: Prompt是怎么设计的？怎么控制幻觉？

**答：**

Prompt设计分几个部分：

**System Prompt**：定义Agent的角色（你是金蝶云星空的结账助手）、能力边界（你只能执行结账相关操作）、行为约束（不确定时必须询问用户确认）

**结账规则Prompt**：定义步骤依赖、前置条件、异常处理策略。这部分是结构化的，不同客户可以配置不同规则。

**幻觉控制的几个手段：**

1. 工具返回值校验：Agent说"折旧计提完成"，我们会校验Tool是否真的返回了成功状态，不是Agent自己编的
2. 状态持久化：每一步的执行状态存在数据库里，不依赖Agent的"记忆"。Agent每次推理前会通过`status_query_tool`获取真实状态
3. 操作确认机制：涉及数据变更的操作（正式结账、损益结转），Agent必须先输出执行计划，经用户确认后才真正执行
4. 输出格式约束：要求Agent以固定JSON格式输出决策，方便程序解析和校验，避免自由文本带来的歧义

## Q6: 结账过程中某一步失败了怎么办？回滚机制怎么设计的？

**答：**

分两个层面：

**Tool层面的失败处理：**
每个Tool内部调用的是ERP业务服务，业务服务本身有事务保证。比如计提折旧失败，该服务内部会回滚，不会产生脏数据。Tool返回失败状态给Agent。

**Agent层面的编排处理：**
Agent收到失败状态后，根据Prompt中定义的策略决策：
- 可重试的错误（比如超时）：自动重试，最多3次
- 业务性错误（比如凭证不平）：暂停流程，通知用户，等待人工处理
- 严重错误（比如服务不可用）：终止整个结账流程，记录断点

**断点续做机制：**
结账进度持久化在数据库中（哪些步骤完成了、哪些失败了）。用户处理完问题后可以说"继续结账"，Agent会从断点处恢复，不需要从头开始。

我们没有做跨步骤的全局回滚，因为ERP结账的每一步都是独立事务，已经成功的步骤（比如折旧已经计提了）不应该因为后面的步骤失败而回滚。这和银行转账不一样，结账更像是一个检查清单，逐项完成。

## Q7: Agent调用LLM的延迟怎么处理？用户体验上怎么优化？

**答：**

这是Agent落地的一个实际痛点。我们的处理方式：

1. **流式输出**：Agent的推理过程用SSE流式返回前端，用户能实时看到Agent在"思考什么"、"正在执行什么"，不会觉得卡住了

2. **异步执行 + 进度推送**：结账本身是个耗时操作（可能几分钟），我们把它设计为异步任务。用户触发后可以去做别的事，通过WebSocket推送进度通知

3. **缓存常见决策路径**：对于标准结账流程（没有异常的情况），Agent的决策路径是固定的。我们会缓存这些路径，减少不必要的LLM调用。只有遇到异常或用户干预时才真正走LLM推理

4. **Tool执行和LLM推理并行**：Tool执行结果返回后，立即触发下一轮LLM推理，不做无谓等待

实测下来，一次完整的自动结账（7个步骤，无异常）大约2-3分钟完成，其中LLM推理占30%左右的时间，主要耗时在业务服务执行上。

## Q8: 并发场景下多个用户同时结账，Agent实例是怎么隔离的？

**答：**

ERP系统是多租户的，结账是按"组织+会计期间"维度的。

**隔离策略：**
- 每次结账请求创建独立的Agent会话（独立的conversationId），上下文完全隔离
- 业务层面加了分布式锁：同一个组织的同一个会计期间，同时只能有一个结账任务在执行。用Redis的SETNX实现，key是`closing:{orgId}:{period}`
- 如果有人正在结账，其他用户触发时会收到提示"当前期间正在结账中，操作人：xxx"

**为什么不用Agent实例池？**
结账是低频操作（一个月一次），不需要维护常驻的Agent实例池。每次创建新会话，结束后释放资源，简单可靠。

## Q9: ERP系统涉及财务数据，Agent的权限怎么控制？

**答：**

这是我们设计时最谨慎的部分：

1. **Tool层权限控制**：Agent能调用哪些Tool，由用户的ERP权限决定。没有结账权限的用户，Agent根本拿不到结账相关的Tool

2. **数据范围控制**：Tool内部调用业务服务时，会带上用户的身份信息和数据权限。Agent不能越权操作其他组织的数据

3. **操作审批机制**：高风险操作（正式结账）需要二次确认。Agent会输出"即将执行正式结账，确认？"，用户确认后才执行

4. **审计日志**：Agent的每一步决策和Tool调用都记录在审计日志中，包括：谁触发的、Agent做了什么决策、调用了哪个Tool、入参出参是什么、最终结果。方便事后审计和问题追溯

5. **能力边界约束**：Prompt中明确定义Agent只能做结账相关操作，不能查询或修改其他模块数据。即使用户通过对话试图让Agent做别的事，Agent会拒绝

## Q10: 技术选型上为什么选择当前方案？对比过哪些？

**答：**

主要对比了三种方案：

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 传统工作流引擎(Flowable) | 成熟稳定、团队熟悉 | 灵活性差、每种流程都要画图维护 | 不适合多变的结账场景 |
| LangChain/LlamaIndex | 生态丰富、社区活跃 | Python生态，和我们Java技术栈不匹配 | 排除 |
| Spring AI + 自研编排 | 和现有Spring Boot体系无缝集成、Java原生 | 相对较新，需要自己处理一些底层问题 | 最终选择 |

选Spring AI的核心原因：
- 金蝶云星空本身是Java/Spring Boot体系，引入Python栈会增加运维复杂度
- Spring AI的ChatClient + Function Calling机制天然支持Tool调用
- 团队Java能力强，能快速上手和排查问题
- 和现有的Spring Security、Spring Data等无缝集成，权限控制和数据访问不需要额外适配

## Q11: 如果让你重新设计，有什么会改进的地方？

**答：**

几个方向：

1. **引入MCP协议**：当时Tool是硬编码注册的，如果用MCP协议，可以实现Tool的动态发现和注册，方便后续扩展更多业务场景（不只是结账，还有对账、报表生成等）

2. **多Agent协作**：目前是单Agent处理所有步骤。如果结账流程更复杂，可以拆成多个专业Agent（凭证检查Agent、折旧计算Agent、结转Agent），由一个编排Agent协调。这样每个Agent的Prompt更聚焦，准确率更高

3. **更好的可观测性**：加入Agent决策链路的可视化追踪，类似分布式链路追踪那样，能看到每一步的推理过程、耗时、Token消耗

4. **离线评估体系**：建立结账场景的测试用例集，每次Prompt调整后自动跑评估，量化准确率变化，而不是靠人工测试

## Q12: 这个项目对你从开发转向架构师有什么启发？

**答：**

最大的启发是：架构设计不只是选技术组件，更重要的是定义系统的边界和交互契约。

具体来说：
- 做KWork时我需要思考Agent层和业务服务层的边界在哪里、怎么解耦。这和微服务拆分的思维是一样的
- 需要考虑非功能性需求：性能（LLM延迟）、安全（权限控制）、可靠性（失败恢复）、可观测性（审计日志）。这些在纯开发阶段很少主动考虑
- 需要做技术选型决策并说服团队，这要求你不只是会用，还要理解每个方案的trade-off
- AI Agent是新技术，没有太多现成的最佳实践，需要自己摸索和沉淀方法论。这种"从0到1定义规范"的能力正是架构师需要的
