## 题目1：多智能体协作架构设计

### 面试官问题
设计一个金融场景的多智能体系统（比如智能投顾），多个Agent怎么协作？怎么拆分？

### 标准答案（STAR格式，口述版）

【S】金融场景需要多个专业能力协同：客户画像分析、市场行情获取、风险评估、产品推荐、合规审查。单一Agent加大量Tool会导致工具选择准确率下降——我在KWork中实测过，超过10个Tool后LLM选错工具的概率显著上升。

【T】设计多Agent协作架构，让每个Agent专注一个领域，通过编排层协调完成复杂任务。

【A】采用"分层编排+专家Agent"模式（类似Crew AI的设计理念，但基于Spring AI实现）：

**架构三层**：
1. 编排Agent（Orchestrator）：接收用户请求，做意图理解和任务分解，将子任务分发给专家Agent
2. 专家Agent集群：每个Agent有独立System Prompt + 专属工具集 + 独立记忆
   - ProfileAgent：客户画像分析（调用CRM系统、交易流水分析）
   - MarketAgent：行情获取与分析（调用行情API、新闻摘要）
   - RiskAgent：风险评估（调用风控模型、信用评分）
   - ProductAgent：产品匹配推荐（调用产品库、适当性匹配）
   - ComplianceAgent：合规审查（规则引擎、监管要求校验）
3. 聚合层：收集各Agent结果，由编排Agent整合为最终回答

**通信模式**：
- 顺序依赖：ProfileAgent到RiskAgent到ProductAgent（风险等级决定可推荐产品范围）
- 并行无依赖：MarketAgent和ProfileAgent可以同时执行
- 投票/验证：ComplianceAgent对最终结果做合规校验（类似guard rail）

**技术实现**：
- 编排Agent做任务分解，用LLM结构化输出TaskPlan（需要哪些Agent、执行顺序、依赖关系）
- CompletableFuture并行调度无依赖的Agent
- 有依赖的串行执行（如RiskAgent依赖ProfileAgent的画像结果）

**共享上下文**：Agent间通过共享黑板（Redis Hash）传递中间结果

【R】相比单Agent+15个Tool，多Agent模式工具选择准确率从82%提升到96%（每个Agent只管3-4个Tool），且各Agent可独立迭代部署。

### 回答要点
1. 先说为什么要拆多Agent（工具太多选择准确率下降）
2. 三层架构：编排到专家到聚合
3. 通信模式：顺序/并行/验证
4. 共享上下文方案（黑板模式）
5. 量化对比：单Agent vs 多Agent

## 题目2：Agent评测体系建设

### 面试官问题
你怎么评估一个Agent系统的效果？怎么保证每次迭代不退步？

### 标准答案（口述版）

**评测体系分三层**：

**一、单元级评测（Tool调用准确率）**
- 构建标注数据集：200+条（用户输入到期望调用的Tool+参数）
- 自动化测试：输入用户query，对比LLM实际选择的Tool和期望Tool
- 指标：Tool选择准确率、参数填充正确率
- 我在KWork中用这个方法把准确率从75%调到98%

**二、任务级评测（端到端完成度）**
- 定义金融场景测试用例
- 评判维度：任务完成率、步骤合理性、事实准确性、合规性
- 人工+LLM as Judge混合评审

**三、安全级评测（对抗测试）**
- 越狱测试：尝试让Agent透露系统Prompt、绕过合规限制
- 幻觉测试：给错误前提，看Agent是否盲信
- 边界测试：极端输入

**回归保障机制**：
代码提交到单元测试到Agent评测套件（200条用例）到准确率>=95%才允许合并到灰度发布10%流量到线上指标监控到全量发布

**持续迭代闭环**：
- 线上bad case自动收集（用户点不满意、对话中断、重试超3次）
- bad case分析到标注为新测试用例到加入评测集
- 评测集只增不删，保证不会旧问题复发

### 回答要点
1. 三层：单元级到任务级到安全级
2. 自动化+人工混合
3. CI/CD集成（准确率门槛）
4. Bad case闭环
5. 结合自己调优75%到98%的经验

## 题目3：LLMOps流水线设计

### 面试官问题
你对LLMOps怎么理解？如果让你建设一个大模型应用的CI/CD流水线，怎么设计？

### 标准答案（口述版）

**LLMOps vs 传统MLOps的区别**：
传统MLOps关注模型训练到评估到部署。LLMOps更关注Prompt管理、Agent编排逻辑、上下文窗口优化、成本控制——因为大部分团队是调用API不是自己训练。

**流水线设计（五个阶段）**：

**1. 开发阶段：Prompt版本管理**
- Prompt存Git，和代码一样版本控制
- System Prompt、Tool description、Few-shot examples全部文件化
- PR review包含Prompt变更的diff

**2. 测试阶段：多维度自动化评测**
- 功能测试：输入到期望输出的断言
- 回归测试：历史bad case不能复现
- 性能测试：延迟P99、Token消耗量
- 安全测试：越狱/注入/信息泄露

**3. 部署阶段：灰度+多版本并行**
- A/B Test：新旧Prompt/模型同时运行，对比效果
- 灰度发布：先10%到50%到100%
- 一键回滚：版本号切换即可回退

**4. 监控阶段：核心指标看板**
- 业务指标：任务完成率、用户满意度
- 技术指标：延迟、Token用量、缓存命中率
- 安全指标：敏感词触发率、越狱尝试次数
- 成本指标：日/周/月Token消耗+预算告警

**5. 反馈阶段：数据飞轮**
- Bad case收集到人工标注到加入训练/微调数据
- Good case也收集到作为few-shot示例
- 周报：模型表现趋势、top问题分类

**结合我的实践**：
TokenUsageAdvisor做用量监控、SemanticCacheAdvisor做成本优化、模型路由做多模型管理——这些就是LLMOps的一部分。

### 回答要点
1. 先区分LLMOps和传统MLOps
2. 五阶段：开发到测试到部署到监控到反馈
3. Prompt版本管理是核心差异点
4. 数据飞轮闭环
5. 结合自己项目的已有实践

## 题目4：金融场景风控Agent设计

### 面试官问题
设计一个实时风控Agent，用户发起转账时要在200ms内决策是放行还是拦截。

### 标准答案（口述版）

**核心挑战**：LLM推理通常需要1-3秒，200ms内不可能走完整的Agent推理链。所以风控Agent不能是纯LLM决策，而是"规则引擎为主+LLM辅助"的混合架构。

**分层决策架构**：

第一层：规则引擎（小于10ms）
- 黑名单命中直接拦截
- 白名单命中直接放行
- 金额小于1000且历史无风险直接放行

第二层：实时风控模型（小于50ms）
- 特征工程：用户画像+交易特征+设备指纹+行为序列
- XGBoost/LightGBM评分模型，返回风险分数
- 分数小于30放行，大于80拦截，30-80进入灰度区间

第三层：LLM Agent（异步，不阻塞主链路）
- 灰度区间的交易进入Agent分析队列
- Agent调用工具：查近期交易模式、对比历史行为、分析收款方信息
- 输出决策建议+理由（给人工审核用）
- 决策延迟容忍5-10秒（先放行+标记可疑，人工复核）

**LLM Agent在风控中的真正价值**：
1. 可疑交易的深度分析（事后/准实时，非实时阻塞）
2. 风控规则的自然语言生成和解释
3. 新型欺诈模式的识别（pattern发现）
4. 人工审核辅助（给审核员摘要+建议）

**一句话总结**：实时风控用规则+轻量模型保证时效，LLM Agent做深度分析保证智能化，两者互补不互斥。

### 回答要点
1. 第一反应：200ms内LLM做不到，必须分层
2. 三层决策：规则到轻量模型到LLM Agent（异步）
3. LLM的真正价值在深度分析而非实时决策
4. 降级策略
5. 金句："规则保时效，Agent保智能，两者互补"

## 题目5：模型部署与推理优化

### 面试官问题
如果要私有化部署一个7B/14B的金融大模型，你会怎么做性能优化？

### 标准答案（口述版）

**部署架构选型**：
- 推理框架：vLLM（PagedAttention，GPU显存利用率最高）或TGI
- 部署方式：Kubernetes + GPU Node Pool + HPA自动扩缩容
- 负载均衡：基于队列深度的路由（不是简单Round Robin）

**推理优化手段（从效果大到小）**：

**1. 量化（降显存+加速）**
- INT8/INT4量化：7B模型从14GB显存降到4-7GB，推理速度提升1.5-2倍
- GPTQ/AWQ量化方案在金融QA场景精度损失小于2%
- 金融领域建议INT8起步，INT4需要评估精度是否够用

**2. KV Cache优化**
- vLLM的PagedAttention：将KV Cache分页管理，避免碎片化
- Prefix Caching：相同System Prompt的请求共享KV Cache

**3. 批处理优化（Continuous Batching）**
- 动态批处理，不同长度请求独立结束不互相等待
- 吞吐量相比静态批处理提升3-5倍

**4. 模型蒸馏（长期方案）**
- 用大模型（70B）生成金融领域的高质量数据
- 蒸馏到7B/14B小模型，特定任务达到大模型90%效果

**5. 推测解码（Speculative Decoding）**
- 小模型草稿+大模型验证，加速1.5-2倍

**监控指标**：TTFT（首Token延迟）、TPS（吞吐量）、GPU利用率（目标70-85%）、队列深度

### 回答要点
1. 推理框架选vLLM并说原因
2. 优化手段按效果排序
3. 量化的精度损失评估（金融场景敏感）
4. 监控指标
5. 展示对实际部署的了解

## 题目6：金融数据安全与隐私保护

### 面试官问题
大模型处理金融数据，隐私安全怎么保障？

### 标准答案（口述版）

**三大风险**：
1. 数据泄露：用户隐私被模型"记住"并在其他对话中泄露
2. 模型攻击：Prompt Injection绕过安全限制
3. 合规风险：个人信息保护法要求数据最小化、可删除

**防护体系（五层）**：

**1. 数据脱敏层（入口）**
- 输入到LLM前自动脱敏：身份证号、银行卡号、手机号替换为占位符
- 正则+NER模型双重检测
- 脱敏映射表存本地，回答时还原

**2. Prompt安全层**
- 输入过滤攻击模式
- 输出过滤系统Prompt泄露
- 角色边界约束

**3. 访问控制层**
- 多租户严格隔离（我的项目已实现：TenantContext+数据过滤）
- RBAC权限
- API密钥管理：轮转+配额+审计

**4. 审计追踪层**
- 所有LLM调用全量日志（脱敏后）
- 异常行为告警
- 对话留痕可追溯

**5. 基础设施层**
- 模型部署内网隔离，不走公网
- 数据不出域
- 定期安全评估

**金融行业特殊要求**：
- 投资建议免责声明自动附加
- 禁止承诺收益率
- 对话记录保留不少于20年（证券监管）
- 数据跨境限制

**金句**：数据不出域是底线，脱敏是常态，审计是保障。

### 回答要点
1. 先列风险再给方案
2. 五层防护都有具体技术手段
3. 结合自己项目已有实践
4. 金融行业特殊合规要求

## 题目7：模型选型与路由策略

### 面试官问题
金融场景下不同任务用不同模型，你怎么做模型路由？

### 标准答案（口述版）

**路由策略（多维度决策）**：

用户请求到意图识别到路由决策：
- 简单FAQ/闲聊到qwen-turbo（快+便宜）
- 复杂分析/推理到qwen-plus/qwen-max（能力强）
- 代码生成/数据分析到deepseek-coder
- 金融专业问题到金融微调模型（自部署）
- 多模态到qwen-vl

**路由依据四维度**：
1. 任务复杂度：简单提取用小模型，推理分析用大模型
2. 延迟要求：实时对话用快模型，离线分析用大模型
3. 成本预算：Token用量监控+每日预算，超额自动降级
4. 安全等级：涉及敏感数据用私有部署模型，通用问题用API

**我的实现经验**（MultiModelConfig）：
- 配置化路由规则，不硬编码
- 模型健康检查：错误率/延迟滑动窗口统计
- 自动降级：主模型异常到备用模型到兜底话术
- A/B测试：新模型灰度对比效果

**金句**：不是选一个"最好"的模型，而是建立模型矩阵——用对的模型做对的事。

### 回答要点
1. 路由策略图
2. 四维度决策
3. 降级链路
4. 结合自己MultiModelConfig实践

## 题目8：RAG在金融场景的挑战

### 面试官问题
RAG用在金融场景有什么特殊挑战？你怎么解决？

### 标准答案（口述版）

**金融RAG三大挑战**：

**挑战1：数据结构复杂**
- 金融文档含大量表格（财务报表）、图表、公式
- 普通文本切分会丢失表格的行列语义
- 方案：表格单独提取为结构化JSON、多模态模型预处理图表

**挑战2：数值精确性要求高**
- 纯RAG检索可能返回多个相似但不同年度的数据，LLM容易搞混
- 方案：元数据精确过滤（先按年度/公司/指标过滤再向量检索）+ 数值校验层 + BM25精确匹配数字

**挑战3：时效性与多版本**
- 同一公司研报每季度更新，旧版本不能当最新数据引用
- 方案：每个chunk带publish_date元数据、检索按时间倒序、过期文档标记archived

**我的实践**：
- 混合检索（向量+BM25+RRF）已实现
- 元数据过滤（tenant_id、document_type）已落地
- 知识库动态更新不用重启

### 回答要点
1. 三个金融特有挑战
2. 每个挑战都有具体方案
3. 结合自己项目RAG实践
4. 数值精确性是金融RAG的命门

## 题目9：Prompt Engineering在金融场景的实践

### 面试官问题
金融场景的Prompt设计有什么特殊考虑？

### 标准答案（口述版）

**金融Prompt四原则**：

**1. 合规约束必须硬编码**
- 绝对不能承诺收益率
- 投资建议必须附加风险提示
- 这些不能靠模型"理解"，要在输出后做规则检查兜底

**2. 专业术语一致性**
- Few-shot示例用标准金融术语
- 数字格式统一：金额带千分位、百分比精确到两位

**3. 推理链显式化（Chain of Thought）**
- 金融决策不能是黑盒，必须让推理过程可追溯

**4. 防幻觉强化**
- 不确定就说不确定，不要编造财务数字

**踩坑经验**：
- 坑1：两个相似工具LLM分不清。解决：Tool description写非常具体的使用条件+入参差异化
- 坑2：模型对数字不敏感。解决：计算逻辑写在Tool里，不让LLM自己算
- 坑3：System Prompt太长指令遵循度下降。解决：核心约束放前面（Primacy Effect）

**金句**：计算交给Tool，决策交给LLM，合规交给规则。

### 回答要点
1. 四原则
2. 至少2个具体踩坑
3. 每个坑的解决方案

## 题目10：智能投顾场景完整方案

### 面试官问题
如果让你设计一个智能投顾Agent，从产品角度讲，它应该能做什么？技术上怎么落地？

### 标准答案（口述版）

**产品能力**：
1. 客户画像分析：根据交易历史+风险问卷，生成投资者画像
2. 资产诊断：分析当前持仓的资产配置合理性
3. 产品推荐：基于画像+市场行情，推荐适配产品
4. 模拟试算：定投/一次性投入的收益模拟
5. 风险提示：市场异动时主动提醒
6. 定期复盘：月度/季度资产报告

**技术架构**：
编排Agent（意图理解+任务路由）
- ProfileAgent（客户画像）：查交易流水、查风险问卷、计算持仓分布
- MarketAgent（市场分析）：获取行情、获取研报摘要、计算指标
- AdvisorAgent（投资建议）：产品匹配、收益试算、适当性校验
- ComplianceAgent（合规审查）：免责声明附加、敏感表述检测

**关键技术点**：
1. 适当性匹配：监管强制要求，产品风险等级不能超过客户风险承受能力
2. 实时行情接入：行情数据TTL极短（分钟级）
3. 回测计算：放在Tool里，不让LLM算
4. 个性化记忆：长期记住客户偏好

**落地节奏**：
- Phase 1（MVP）：FAQ问答+RAG产品知识库（2周）
- Phase 2：接入客户数据+资产诊断（4周）
- Phase 3：多Agent协作+产品推荐（6周）
- Phase 4：主动服务（8周）

**金句**：AI投顾不是替代人工顾问，是让每个客户都能获得专业服务。

### 回答要点
1. 先从产品角度讲能力
2. 再讲技术架构（多Agent）
3. 合规适当性是命门
4. 落地节奏分Phase

## 题目11：你对金融大模型未来方向的判断

### 面试官问题
你觉得大模型在金融领域未来1-2年最有价值的方向是什么？

### 标准答案（口述版）

**三个方向最有确定性价值**：

**1. 智能化运营（降本增效，最快落地）**
- 客服自动化、单据生成、知识管理
- 价值：直接可量化的人力成本下降

**2. 辅助决策（增量价值，需要积累）**
- 信贷审批辅助、投研助手、风控增强
- 价值：决策质量提升

**3. 个性化服务（长期方向，壁垒最深）**
- 智能投顾、精准营销、主动服务
- 价值：提升客户体验和留存

**为什么不是AI交易**：监管限制大、高频交易不需要LLM、可解释性要求和LLM黑盒属性冲突

**我的判断**：短期看运营降本（6个月见效），中期看辅助决策（1-2年构建数据壁垒），长期看个性化服务（2-3年形成竞争力）。Agent多智能体是实现这些的核心技术载体。

### 回答要点
1. 给出明确判断
2. 按时间线排序
3. 解释为什么不看好某些方向（展示独立思考）
4. 落脚到多智能体（和岗位JD呼应）

## 题目12：你为什么适合这个岗位？

### 面试官问题
结合你的经验，为什么觉得自己适合做金融大模型多智能体应用？

### 标准答案（口述版，2分钟）

**三个匹配点**：

**1. Agent系统实战经验**
独立设计开发了KWork智能结账Agent——Spring AI + ReAct模式 + MCP协议 + 多Tool编排。从Prompt设计、工具注册、防幻觉、到模型路由降级，全链路踩过坑。

**2. 金融业务理解**
在数字银行做过信用卡还款系统——支付幂等、状态机、分布式事务、对账。在金蝶做过资产管理折旧计算——大数据量批处理、多租户SaaS。金融业务的核心（准确性、安全性、合规性）我在实际项目中体会过。

**3. 工程化能力**
不只是能让Agent跑起来，更关心怎么让它稳定运行在生产环境——多模型路由、Token成本控制、语义缓存优化、限流降级、可观测性监控。这些正是LLMOps需要的能力。

**我还想做的**：
- 多Agent协作（从单Agent到Agent团队）
- Agent评测体系化（从手动验证到CI/CD自动化回归）
- 金融专属RAG优化（表格处理、数值精确性、时效性管理）

这些正好是这个岗位的核心方向。

### 回答要点
1. 三个匹配点对应JD三个关键词
2. 每个点有具体项目支撑
3. 最后说"还想做什么"展示成长方向
4. 控制在2分钟内

## 二面通用策略

### 和一面的区别
- 一面考知识广度（八股+项目经历），二面考深度和判断力
- 二面面试官通常是技术Leader或架构师，更关注"为什么"而非"怎么做"
- 会出开放式设计题，没有标准答案，看你的思考框架

### 回答技巧
1. 先给结论再展开：不要铺垫太长，30秒内说清楚你的方案选择
2. 主动说trade-off：每个方案都有代价，你主动说比被面试官问出来好
3. 用数字说话：延迟多少ms、准确率多少、成本降多少
4. 关联岗位JD：回答时自然带出"多智能体"、"MLOps"、"风控"等关键词
5. 承认不足+有想法：不会的不要编，说"这块我还没深入实践，但我的思路是..."

### 可能的反问（你问面试官）
1. "团队目前多智能体的技术选型是什么？自研框架还是基于开源？"
2. "金融场景Agent上线后最大的挑战是什么？准确率还是延迟还是合规？"
3. "评测体系目前是怎么做的？自动化程度如何？"
4. "模型是自研微调还是调用API为主？未来方向是什么？"
