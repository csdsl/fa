# 面试训练记录 - Day4

## 题目1：线程池核心参数设定

### 面试官问题
你在金蝶做资产折旧计算优化时，ThreadPoolExecutor的核心参数是怎么设定的？corePoolSize、maximumPoolSize、队列容量分别是多少，为什么这样设？

### 标准答案（口述版）

线程池参数设定我分开发期和压测调优两个阶段来说。

开发期，先用经典公式：CPU密集型任务取 N+1，我们服务器8核，初始设corePoolSize=9。但这只是起点。

数据分片策略是按资产类别+组织维度分片，每批2000张卡片，50万张大约产生250个计算任务。所以队列容量我设256，刚好覆盖所有分片任务，不需要设很大——队列太大反而会掩盖消费能力不足的问题。

压测阶段做了几轮调整：先把core调到16，发现CPU持续在85%以上，GC频率也上来了，说明线程争抢太激烈。回退到12，CPU稳定在60-70%，吞吐量反而更好——因为减少了上下文切换开销。最终参数：core=12，max=20，队列256，拒绝策略用CallerRunsPolicy，让提交线程自己执行，起到天然限流的效果。

为什么不用更大的队列？因为资产折旧是纯计算型任务，瓶颈在CPU不在IO，堆积更多任务没意义，不如控制并发度让每个任务更快完成。

### 回答要点
1. 分阶段说：开发期公式→压测调优最终值
2. 队列容量要关联分片策略（250个任务→256队列）
3. 必须说拒绝策略（CallerRunsPolicy）
4. 解释为什么CPU密集型不用大队列
5. 有压测数据支撑（CPU 85%→60-70%）

## 题目2：分片异常处理与重试机制

### 面试官问题
如果某个分片计算失败了，你怎么处理？会影响其他分片吗？失败的分片怎么重试？

### 标准答案（口述版）

异常处理我设计了三层机制：隔离、重试、兜底。

**隔离层**：每个分片任务用独立的try-catch包裹，一个分片抛异常不会影响其他分片的执行。我用CompletableFuture提交任务，通过exceptionally回调捕获异常，不会导致整个批次中断。

**重试层**：失败的分片会进入重试队列，最多重试3次，每次间隔递增（1s、3s、5s）。这里我没用Spring Retry，因为需要自定义重试粒度——只重试失败的那一批卡片，不是整个方法重试。实现上就是一个带计数器的while循环 + Thread.sleep延迟。

**兜底层**：3次重试都失败的分片，将失败的卡片ID、异常原因、时间戳写入一张`depreciation_fail_log`表。同时标记这批卡片的折旧状态为"待处理"。系统会推送告警给财务人员，他们修正数据后可以在界面上对失败的卡片单独触发重算——不需要重跑整个50万张。

**并发安全**：每个分片写入结果时是按卡片ID维度更新的，分片之间没有数据交集，天然不冲突。最终结果汇总用CountDownLatch等待所有分片完成，再统一更新批次状态。

这个设计的好处是：成功率99.5%以上的卡片不受少量异常数据拖累，失败影响面从"整批回滚"缩小到"单卡片级"。

### 回答要点
1. 三层结构：隔离→重试→兜底
2. 说清楚技术实现：CompletableFuture + exceptionally
3. 重试策略：3次+递增间隔，不用Spring Retry的原因
4. 失败持久化：写fail_log表 + 支持局部重算
5. 并发安全：按卡片ID维度天然不冲突
6. 量化结果：影响面缩小99%

## 题目3：防重复扣款（幂等性设计）

### 面试官问题
信用卡还款场景中，你是怎么防止重复扣款的？如果用户短时间内多次点击还款按钮，你的方案怎么保证只扣一次？

### 标准答案（口述版）

防重复扣款我们做了前端+后端+数据库三层防护，是银行级的幂等方案。

**第一层：前端防抖**。用户点击还款按钮后立即置灰，防止物理层面的重复提交。但这只是体验优化，不是安全保障。

**第二层：后端幂等Token**。用户进入还款页时，后端生成一个一次性的repayToken存到Redis（TTL 5分钟）。提交还款时必须携带这个token，后端用`SETNX`或`DEL`原子操作校验——token存在则放行并删除，不存在则拒绝。这保证同一个还款意图只能提交一次。

**第三层：数据库唯一约束**。每笔还款生成唯一的交易流水号，规则是`userId + billMonth + timestamp + sequence`。还款记录表对这个流水号加唯一索引，即使并发穿透了Redis层，数据库也能兜底。

**状态机控制**：还款订单有严格的状态流转——`CREATED → PAYING → SUCCESS/FAILED`。状态变更用乐观锁实现：
```sql
UPDATE repayment SET status='PAYING' 
WHERE id=? AND status='CREATED'
```
只有affected rows=1才继续调用银行扣款接口。如果是0说明已经有其他请求在处理了，直接返回"处理中"。

**异常场景**：如果调了银行扣款接口但没收到响应（超时），不会重复调用，将状态置为`UNCERTAIN`，由定时对账任务去银行侧查询真实结果。

上线后还款重复扣款投诉从每月几单降到零。

### 回答要点
1. 三层防护：前端→后端Redis Token→数据库唯一约束
2. 状态机 + 乐观锁SQL是核心
3. 超时场景单独处理（UNCERTAIN状态）
4. 有量化结果（投诉降到零）
5. 口述至少2分钟，不能一句话带过

## 题目4：银行扣款超时处理

### 面试官问题
调了银行扣款接口但网络超时了——既不知道扣成功了还是失败了，你怎么处理？

### 标准答案（口述版）

银行扣款超时是支付系统最经典的不确定状态问题，我们采用"状态悬挂+双通道确认"的方案。

**第一步：立即标记为UNCERTAIN状态**。调银行接口超时后，不能假设成功也不能假设失败，将这笔订单状态设为UNCERTAIN，同时给用户返回"还款处理中，请稍后查看结果"。

**第二步：双通道确认**。两条路同时走：
- **主动查询**：将这笔UNCERTAIN订单的流水号写入本地消息表，由定时任务（每30秒扫描一次）拿着原始交易流水号调银行的"交易结果查询接口"。注意不是重新扣款，是查询之前那笔的结果。
- **被动通知**：银行侧也有异步通知机制（webhook），扣款完成后会回调我们的通知接口。收到通知后验签、更新状态。

两条路谁先返回就以谁为准，通过流水号幂等保证不会重复更新。

**第三步：重试与升级**。主动查询最多重试5次，间隔递增（30s、1min、2min、5min、10min）。如果5次都查不到结果，自动升级为人工工单，由运营人员联系银行确认。

**第四步：对账兜底**。每天凌晨跑T+1对账批处理，拉取银行的扣款明细和本地还款记录做双向对账。UNCERTAIN状态超过24小时还未确认的，对账任务会最终裁定。

**为什么用本地消息表而不是直接发MQ？** 因为要保证"状态更新"和"消息投递"的原子性。先写本地消息表（和业务表在同一个事务），再由定时任务投递，避免消息丢失。

核心原则是：**宁可多查不可多扣，宁可慢不可错**。

### 回答要点
1. 第一反应：标记UNCERTAIN，不假设成功/失败
2. 双通道：主动查询 + 被动webhook
3. 递增重试 + 人工升级
4. T+1对账兜底
5. 解释为什么用本地消息表
6. 金句："宁可多查不可多扣，宁可慢不可错"

## 题目5：Agent防幻觉机制

### 面试官问题
KWork用了ReAct模式，如果LLM产生幻觉——比如它"认为"折旧已经算完了但实际上没有——你怎么防止？Tool返回结果有没有做校验？

### 标准答案（口述版）

防幻觉我们从三个层面做了控制：

**工具层：强制校验，不信LLM的"判断"只信Tool的"返回"**
- 入参校验：Tool的参数有严格的JSON Schema定义，类型不对、必填缺失直接拒绝调用，错误返回给LLM要求修正
- 出参校验：Tool返回结果做格式和业务规则校验（比如折旧金额不能为负、累计折旧不超过原值-残值），校验失败标记为无效结果
- 核心原则：LLM说"折旧已完成"没用，必须调confirmDepreciation工具，系统查数据库返回真实状态

**推理层：约束ReAct循环的行为边界**
- Prompt里明确写了：你不能假设任何步骤已完成，必须调用对应Tool获取实际状态
- 每一步Observation后加入反思环节：如果中间结果异常（如试算不平衡），提示LLM回溯检查之前的步骤
- 工具描述里写明前置条件："调用closePeriod前，trialBalance必须返回balanced=true"

**系统层：硬性兜底，防止失控**
- 最大ReAct迭代次数设为10步，超过强制终止，返回"无法自动完成，需人工介入"
- 单次对话的Tool调用次数也有上限，防止无限重试
- 所有Tool调用记录审计日志，方便事后追溯Agent的决策路径

这套机制上线后，Agent误操作率为零——因为它根本没有能力绕过Tool做任何实际业务操作。

### 回答要点
1. 三层结构：工具层→推理层→系统层
2. 核心金句："不信LLM的判断，只信Tool的返回"
3. 具体例子：confirmDepreciation、trialBalance
4. 最大迭代次数 + 强制终止
5. 量化结果：误操作率为零

## 题目6：RabbitMQ消息可靠性

### 面试官问题
你用RabbitMQ做短信异步发送，如果消息丢了怎么办？怎么保证消息的可靠投递？

### 标准答案（口述版）

RabbitMQ消息可靠性我们做了三层保障加一个兜底：

**生产端：Confirm + 本地消息表**
- 开启Publisher Confirm模式，消息到达Exchange后回调ack
- 配置Return回调：消息无法路由到队列时触发returnedMessage回调
- 发送前先写本地消息表（状态=SENDING），收到ack后更新为SENT。定时任务每30秒扫描超时未确认的消息重新投递，最多重试3次

**MQ层：持久化 + 高可用**
- Exchange和Queue都设置durable=true，消息设置deliveryMode=2（持久化）
- 集群用镜像队列（后来升级到Quorum Queue），消息至少在2个节点持久化后才返回ack

**消费端：手动ACK + 死信 + 幂等**
- 关闭autoAck，业务处理成功后手动basicAck
- 处理异常时basicNack，requeue=true重回队列。设了最大重试次数（header里记录retry count），超过3次进入死信队列（DLX），告警系统通知运维
- 幂等控制：每条消息有唯一messageId，消费前用Redis SETNX（key=msg:{id}，TTL=24h）判重，防止重复发短信

**兜底：对账**
- 每日定时任务扫描本地消息表中状态为SENDING超过1小时的记录，与短信通道商的发送记录做对账，补发遗漏的

**短信场景特殊处理**：接了两家短信通道（阿里云+腾讯云），主通道发送失败自动切备用通道，保证验证码类短信到达率>99.5%。

上线后消息丢失率从万分之三降到零投诉。

### 回答要点
1. 先给结论框架："三层保障+一个兜底"
2. 生产端：Confirm + Return + 本地消息表
3. MQ层：持久化 + 镜像队列/Quorum Queue
4. 消费端：手动ACK + 死信队列 + 幂等SETNX
5. 兜底：每日对账
6. 场景关联：双通道短信降级
7. 量化：丢失率降到零

## 题目7：多租户SaaS数据隔离方案

### 面试官问题
假设让你从零设计一个多租户SaaS系统的数据隔离方案，你会怎么做？结合金蝶ERP经验。

### 标准答案（口述版，STAR格式）

【S】金蝶云星空旗舰版是面向中大型企业的ERP SaaS，客户既有年营收百亿的集团企业，也有几千人的中型公司。不同客户对数据隔离的要求差异很大。

【T】我参与的资产管理模块需要适配平台的多租户架构，目标是同一套代码支撑所有租户，同时满足不同隔离等级。

【A】采用"混合隔离"策略：

三种隔离级别：
| 方案 | 隔离度 | 成本 | 适用场景 |
|------|--------|------|----------|
| 独立数据库 | 最高 | 最高 | 金融、政企大客户 |
| 共享库独立Schema | 中等 | 中等 | 中型客户 |
| 共享表tenant_id | 最低 | 最低 | 海量中小租户 |

具体实现：
1. **逻辑层统一**：所有表有tenant_id字段，MyBatis拦截器自动注入租户条件
2. **租户上下文**：请求进来从JWT解析tenantId存入ThreadLocal（TenantContext）
3. **大客户升级**：动态数据源路由（AbstractRoutingDataSource），根据tenantId切换专属库
4. **安全兜底**：API层权限校验 + 定期审计是否有SQL绕过拦截器

【R】同一套代码支撑200+租户。大客户（~10%）独立库满足等保三级，90%共享库方案单库成本降低80%。租户间数据泄露：零。

**追问应对**：
- 为什么不全用独立库？→ 200个RDS实例成本爆炸，90%租户数据量小
- 为什么不全用共享表？→ 金融/政企客户合规审计要求物理隔离
- 所以是分级策略，按付费等级和合规要求选择

### 回答要点
1. 先说三种方案对比（表格思维）
2. 再说金蝶实际用的是混合策略
3. 技术实现：TenantContext + MyBatis拦截器 + AbstractRoutingDataSource
4. 为什么这么选（成本 vs 合规的trade-off）
5. 量化：200+租户、成本降80%、泄露为零

## 通用面试技巧总结

1. **先给结论框架，再展开细节**："我们从三个层面来保障..."
2. **项目题必用STAR**：一句话背景→职责→分步方案→量化结果
3. **每个方案都要说"为什么"**：不是只说做了什么，要说为什么选这个不选那个
4. **带数字**：提速1.6倍、投诉降到零、成本降80%、CPU 60-70%
5. **口述控制在2-3分钟**：太短显得没深度，太长面试官会打断
