## 题目1：为什么用AI Agent做结账，传统工作流引擎不行吗？

### 面试官问题
KWork为什么不用传统工作流引擎（Activiti/Flowable）来做结账？Agent的优势是什么？

### 标准答案（口述版）

金蝶云星空的期末结账是多模块、多步骤、有依赖关系的复杂流程：凭证检查→调汇→计提折旧→费用摊销→损益结转→对账→正式结账。

传统工作流引擎的痛点：
- 不同企业结账流程有差异，每种都要画流程图维护
- 异常处理只能走预设分支，不够灵活
- 用户想跳过某步或重试某步，改起来麻烦

Agent的优势：
- 通过Prompt定义规则，不同客户只需调整Prompt而非重画流程
- Agent能根据当前状态动态决策，遇到异常自主判断是跳过、重试还是告警
- 支持自然语言交互，用户可以说"跳过调汇直接计提"

一句话总结：传统工作流适合确定性流程，Agent适合有判断、有变化的半结构化流程。结账恰好是后者。

### 回答要点
1. 先说结账流程的复杂性（7个步骤+依赖关系）
2. 传统工作流的3个痛点
3. Agent的3个优势
4. 金句："确定性流程用工作流，半结构化流程用Agent"

## 题目2：KWork的整体技术架构

### 面试官问题
KWork智能结账Agent的整体架构是怎样的？用了什么LLM、什么框架？

### 标准答案（口述版）

四层架构：

1. **交互层**：结账面板+对话框，支持一键自动结账和对话式交互
2. **Agent编排层**：基于Spring AI的ChatClient，采用ReAct模式（推理→调用工具→观察结果→决定下一步）。LLM用的是通义千问qwen-plus（公司和阿里云合作，内网专线，数据不出域）
3. **工具层**：通过MCP协议注册各结账工具——check_voucher_tool、depreciation_tool、exchange_adjustment_tool、profit_loss_carry_tool、period_close_tool、status_query_tool
4. **业务服务层**：Tool调用已有的ERP业务服务API，保持事务性和权限控制不变

调用链路：用户触发→Controller→ChatClient发起对话→LLM推理决定调用哪个Tool→通过MCP调用结账服务→返回结果→LLM判断下一步→循环直到完成

选Spring AI的原因：金蝶是Java/Spring Boot体系，引入Python栈会增加运维复杂度，且Spring AI和现有Spring Security、Spring Data无缝集成。

### 回答要点
1. 四层分明：交互→编排→工具→业务服务
2. 必须提到LLM选型和原因
3. 说清楚ReAct循环
4. 强调Tool只是编排者，不绕过业务规则
5. 解释技术选型决策（为什么Spring AI不用LangChain）

## 题目3：MCP vs Function Calling的区别

### 面试官问题
MCP解决什么问题？和Function Calling有什么区别？

### 标准答案（口述版）

MCP在我们场景里解决的问题：
- 金蝶云星空是多模块ERP，后续要扩展到对账、报表、资产盘点等场景
- 用MCP后，每个业务模块作为独立MCP Server暴露Tool，Agent作为Client动态发现和调用
- 新增场景只需部署新的MCP Server，Agent不用改代码

和Function Calling的区别：
- Function Calling：Tool和Agent在同一进程内，编译时绑定，简单直接，适合工具少且固定的场景
- MCP：Tool和Agent可跨进程/跨服务，标准协议通信，支持动态发现、版本管理、权限隔离，适合平台化多团队协作

一句话：Function Calling是"内聚"，MCP是"解耦"。选MCP因为金蝶是平台型产品，需要各模块独立演进。

### 回答要点
1. 先说MCP在你场景中解决什么问题
2. 对比两者：进程内vs跨进程、编译时vs运行时、固定vs动态发现
3. 金句："Function Calling是内聚，MCP是解耦"
4. 结合业务：平台型产品多团队协作需要解耦

## 题目4：线程池参数设定

### 面试官问题
50万资产卡片折旧计算的ThreadPoolExecutor参数怎么设定的？

### 标准答案（口述版）

分开发期和压测调优两阶段：

开发期：CPU密集型取N+1，8核服务器初始core=9。分片策略是每批2000张，50万产生约250个任务，队列容量设256。

压测调优：先把core调到16，CPU持续85%以上+GC频率升高，说明争抢太激烈。回退到12，CPU稳定60-70%，吞吐反而更好（减少上下文切换）。

最终参数：core=12，max=20，队列256，拒绝策略CallerRunsPolicy。

为什么不用ForkJoinPool：批量独立任务没有父子关系，ThreadPoolExecutor更直观可控。
为什么不用更大队列：纯计算型任务瓶颈在CPU不在IO，堆积任务没意义。

### 回答要点
1. 分阶段说（公式→压测→最终值）
2. 所有参数都要有理由
3. 队列容量关联分片策略
4. 必须说拒绝策略
5. 压测数据：CPU 85%→60-70%

## 题目5：RabbitMQ消息可靠性

### 面试官问题
怎么保证RabbitMQ消息不丢失？

### 标准答案（口述版）

三个环节分别保障，加一个兜底：

**生产端→Broker**：Publisher Confirm + Return回调，confirm失败记录本地消息表定时重试。

**Broker存储**：Exchange/Queue设durable持久化，消息deliveryMode=2，镜像队列集群模式（或升级到Quorum Queue）。

**Broker→消费端**：关闭自动ACK改手动，处理成功才basicAck。异常basicNack+requeue，超过3次进死信队列DLX告警。消费端幂等：消息ID + Redis SETNX判重。

**兜底**：本地消息表 + 每日对账。

### 回答要点
1. 先给框架："三个环节+一个兜底"
2. 每个环节都要有具体技术手段
3. 消费端必须提到死信队列和幂等
4. 兜底是本地消息表对账

## 题目6：短信重复消费防护

### 面试官问题
短信场景下怎么防止消息重复消费导致用户收到多条短信？

### 标准答案（口述版）

三层防护：

1. **Redis快速去重**：消费者拿到消息后用MessageID做SETNX（TTL 24h），SET失败直接ACK丢弃
2. **数据库兜底**：短信发送记录表加唯一索引(用户ID+业务类型+业务流水号)，插入成功才调短信网关
3. **网关层面**：对接的短信网关支持相同手机号+相同内容N分钟内去重

为什么不只用数据库：Redis判重O(1)毫秒级返回，大部分重复在Redis层拦掉，不打数据库。

### 回答要点
1. 三层：Redis→数据库唯一索引→网关去重
2. 解释为什么需要Redis层（性能）
3. 具体的Redis key设计和TTL

## 题目7：分布式多维度限流方案

### 面试官问题
你们的限流方案是怎么设计的？支持哪些维度？

### 标准答案（口述版）

三层架构：
1. **Nginx层**（IP级粗粒度）：limit_req_zone按IP限流，防攻击和爬虫
2. **API网关层**（接口级）：Redis + Lua脚本实现滑动窗口，Key=`rate_limit:{接口路径}:{时间窗口}`
3. **业务层**（用户级）：Key=`rate_limit:{userId}:{接口路径}:{时间窗口}`

核心实现：Redis Sorted Set + Lua脚本（滑动窗口算法），保证原子性。

限流规则配置化（Nacos），支持动态调整。被限流返回429 + Retry-After头。

高可用：Redis集群，Redis不可用时降级为本地Guava RateLimiter。

算法选择：银行/ERP用滑动窗口（严格控速），秒杀用令牌桶（允许突发）。

### 回答要点
1. 三层：Nginx→网关→业务
2. 具体的Key设计
3. 算法：滑动窗口 + Redis Lua
4. 高可用降级：Redis挂了用本地Guava
5. 算法选择依据

## 题目8：ELK日志架构与冷热分离

### 面试官问题
日志量大了ES查询变慢，你们怎么优化？冷热分离怎么做？

### 标准答案（口述版，STAR格式）

【S】数字银行App 20多个微服务，日均日志量30GB，累积2TB+，近期查询从毫秒退化到5-8秒。

【T】优化ES日志架构，目标近7天日志查询<1秒，控制存储成本。

【A】三件事：
1. **按天建索引 + ILM生命周期管理**：Hot（0-7天，SSD，1副本）→ Warm（7-30天，HDD，0副本，force merge）→ Delete（>90天）。通过node.attr.box_type标记节点角色。
2. **索引模板优化**：热索引5分片1副本，温索引shrink到2分片。Mapping优化：message字段做keyword不做全文分析，日志级别/服务名用keyword类型。
3. **查询层优化**：Kibana默认查询改为最近24小时，按traceId加routing，禁止深度分页（超10000用search_after）。

【R】近7天查询P99从8秒降到400ms。存储成本降40%。CPU利用率从75%降到45%。

### 回答要点
1. 用STAR格式，先说数据量和痛点
2. 冷热分离关键：硬件差异（SSD vs HDD）+ ILM自动迁移
3. 三个优化维度：存储策略+索引设计+查询优化
4. 量化结果必须有

## 题目9：资产管理模块分表方案

### 面试官问题
资产管理模块数据量大了怎么分表？分片Key怎么选？

### 标准答案（口述版）

方案：按租户ID分库，库内按资产类别分表。

**为什么按组织ID分库**：ERP查询100%带组织ID条件，折旧计算按组织批量处理，分库后是单库操作事务简单。

**库内分表**：
- 资产卡片表按资产类别水平分表（固定资产、无形资产、长期待摊）
- 折旧明细表按年月分区（增长最快，查询都带期间条件）

**中间件**：ShardingSphere-JDBC

**注意事项**：
- 跨组织报表走异步汇总
- 全局唯一ID用雪花算法
- DDL变更需批量执行

核心原则：分片Key跟着业务最高频查询条件走。

### 回答要点
1. 分片Key选择依据（最高频查询条件）
2. 分库和分表是两个维度
3. 中间件选型
4. 跨分片查询怎么解决
5. 金句："分片Key跟着业务最高频查询条件走"

## 题目10：信用卡还款的分布式事务方案

### 面试官问题
还款涉及多个服务（扣余额+更新账单+发通知），分布式事务怎么做？

### 标准答案（口述版）

推荐方案：本地消息表 + 最终一致性（不用TCC）。

**为什么不用TCC/Seata**：TCC开发成本高且外部渠道不一定支持Cancel，全局锁性能差。还款是单向推进流程（扣款成功→更新账单→发通知），不需要全局回滚，允许秒级延迟。

**流程**：
1. 账户服务开启本地事务：扣余额 + 写本地消息表（同一事务保证原子性）
2. 定时任务/投递器扫描消息表，发到RabbitMQ
3. 信用卡服务消费消息，减欠款更新账单
4. 通知服务消费消息，发短信

**可靠性保障**：定时任务30s扫描PENDING消息重投，最大重试5次超过告警。消费端用msg_id + Redis做幂等。

**对比TCC的适用场景**：跨行实时转账（A扣B加必须原子性）适合TCC，还款这种单向流程用最终一致性更合适。

### 回答要点
1. 先说方案选择及理由（为什么不用TCC）
2. 核心是本地消息表保证原子性
3. 完整流程：扣款→消息表→MQ→消费
4. 可靠性：重试+幂等
5. 对比TCC适用场景

## 题目11：多线程结果收集+异常处理

### 面试官问题
CompletableFuture怎么收集50万条折旧计算的结果？部分失败怎么办？

### 标准答案（口述版）

用CompletableFuture提交和收集：
1. 数据分片（每批2000条）
2. CompletableFuture.supplyAsync提交到自定义线程池
3. CompletableFuture.allOf等待全部完成
4. 逐个get()区分成功和失败
5. 失败批次记录后重试（最多3次）

**异常处理策略**：部分失败不影响整体。资产折旧按卡片独立计算，A失败不影响B。成功的正常入库，失败的标记"计算异常"人工处理。

**为什么用CompletableFuture而非Future**：支持allOf组合、exceptionally异常处理、thenApply链式处理，且可指定自定义线程池（不用commonPool）。

### 回答要点
1. 分片→提交→allOf等待→逐个收集
2. 部分失败不回滚整体
3. 和Future的区别（allOf + exceptionally + 自定义线程池）

## 题目12：5000租户批量报表架构设计

### 面试官问题
5000租户每月生成报表，每个30秒，怎么在6小时内完成？

### 标准答案（口述版）

先算账：5000×30秒=41.7小时串行不可能。6小时完成需至少8并发，设计16并发留余量。

架构：
1. **调度层**：XXL-Job每月1号触发，主任务拆分5000个子任务写入任务表
2. **分发层**：数据库任务表(status/worker_id/retry_count)，多Worker抢占式消费（SELECT FOR UPDATE或Redis队列）
3. **执行层**：4台机器×4线程=16并发，每个Worker取任务→计算→生成文件→更新状态
4. **失败重试**：FAILED且retry<3自动重置为PENDING；Worker心跳超时5分钟自动恢复
5. **进度监控**：按status统计接入Grafana看板，失败率超阈值告警
6. **资源保护**：数据库连接池限制、大租户优先、限流保护在线业务

### 回答要点
1. **先算账**——这是架构设计题的标准开场（量化分析）
2. 调度→分发→执行→监控完整链路
3. 失败重试和超时恢复
4. 资源隔离保护线上业务
